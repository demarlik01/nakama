## 1. Problem Statement

### 현재 상황

팀에서 AI를 활용하는 방식은 크게 두 가지다:

**A. 개인 도구형** — 각자 Copilot, Claude, ChatGPT 등을 개별적으로 사용
- 팀 컨텍스트 없음 (프로젝트 히스토리, 의사결정 배경을 매번 설명)
- 사용 편차 큼 (잘 쓰는 사람과 아예 안 쓰는 사람)
- 지식이 개인에게 고립됨 (AI와의 대화가 팀에 공유되지 않음)

**B. 워크플로우 자동화형** — 특정 트리거에 AI 단계 추가
- 정해진 트리거 → 정해진 동작 (유연성 없음)
- 설정이 복잡하고 유지보수 비용 높음
- "자동화"이지 "지능"이 아님

### 핵심 문제

> **AI를 "도구"가 아닌 "팀원"으로 쓸 수 있는 방법이 없다.**

- 팀의 맥락을 이해하고
- Slack에서 자연스럽게 소통하며
- 시키지 않아도 할 일을 찾아서 하는

그런 AI 팀원은 아직 없다.

### 기존 솔루션의 한계

| 솔루션 | 문제 |
|--------|------|
| GitHub Copilot | 코드 자동완성에 한정, 팀 컨텍스트 없음 |
| ChatGPT/Claude (웹) | 매번 컨텍스트 설명, 도구 접근 제한, 팀과 분리 |
| Devin / Cursor | 개인 개발 도구, 팀 협업 레이어 없음 |
| Custom Slack Bot | 대부분 Q&A 수준, 능동적 행동 불가 |
| n8n / Zapier + AI | 워크플로우 자동화, 판단력 없음 |

## 2. Solution

### 한 줄 요약

> **Slack에서 소통하고, 팀 맥락을 이해하며, 능동적으로 일하는 AI 팀원을 쉽게 추가한다.**

### 핵심 가치 (3가지)

1. **에이전트를 쉽게 띄울 수 있다** — 역할 정의 + Slack 연결이면 끝
2. **Slack으로 에이전트와 쉽게 소통한다** — DM, 멘션, 스레드로 자연스러운 대화
3. **에이전트는 팀 맥락을 이해하고 능동적으로 일한다** — 기억을 가지고, 행동 지침에 따라, 시키지 않아도 할 일을 찾아서 한다

### 핵심 컨셉

**"워크플로우 자동화가 아닌 팀원 추가"**

Nakama은 자동화 도구가 아니다. 역할에 관계없이 다음과 같은 특성을 가진 **AI 팀원**이다:

- AGENTS.md에 "개발자"라고 쓰면 개발자
- "마케터"라고 쓰면 마케터
- "CS 담당"이라고 쓰면 CS 담당
- 각 팀의 필요에 맞게 에이전트를 정의하고 띄울 수 있음

## 3. Target User

### Primary: 팀 리더 / 운영 담당자
- 팀 생산성 향상에 관심
- AI 도입을 검토 중이지만 "챗봇 수준"에 실망
- Slack 중심 워크플로우
- 직군 무관 (개발, 마케팅, CS, 운영 등)

### Secondary: 팀원 (IC)
- 반복 작업 부담
- AI에 팀 컨텍스트를 매번 설명하는 게 귀찮음
- "그냥 알아서 해줬으면"

## 4. Core Value Propositions

### V1: 에이전트를 쉽게 띄울 수 있다

> "새 에이전트 추가하는 데 5분."

**에이전트 생성 플로우:**
1. Web UI에서 "새 에이전트" 클릭
2. 이름 입력 (예: "코드리뷰어", "마케팅 어시스턴트")
3. 행동 지침 작성 (역할 템플릿에서 시작하거나 직접 작성)
4. Slack 채널 연결
5. → 에이전트 시작

**에이전트 정의 구조:**
Web UI에서 입력한 내용은 워크스페이스에 마크다운으로 저장된다:

| 파일 | 역할 | 생성 주체 |
|------|------|----------|
| `AGENTS.md` | 역할, 톤, 행동 규칙 | **사용자** (Web UI 또는 직접 편집) |
| `MEMORY.md` | 장기 기억 | 에이전트 (자동 축적) |
| `TOOLS.md` | 도구 접근 정보, 메모 | 에이전트 (필요 시 자동 생성) |
| `HEARTBEAT.md` | 능동적 행동 체크리스트 | 사용자 또는 에이전트 |

사용자가 직접 쓰는 건 **AGENTS.md 하나**. 나머지는 시스템이 자동 생성하거나 에이전트가 스스로 관리한다.

고급 사용자는 마크다운 파일을 직접 편집 가능.

### V2: Slack-native 소통

> "슬랙에서 멘션하면 끝. 별도 도구 안 열어도 됨."

- 기존 워크플로우(Slack) 안에서 동작 — 학습 비용 제로
- DM, 멘션, 스레드 지원
- 채널에서 팀 전체가 AI와의 대화를 볼 수 있음 (지식 공유)

### V3: 팀 맥락 + 능동적 행동

> "3개월 전 의사결정 배경을 설명 안 해도 알아. 시키지 않아도 할 일을 찾아서 해."

- MEMORY.md에 컨텍스트 축적 (에이전트가 스스로 관리)
- 행동 지침(AGENTS.md)에 따라 판단하고 능동적으로 움직임
- 하트비트/스케줄러로 주기적 체크 → 할 일 발견 시 행동

### V4: 간편한 관리

> "Web UI에서 에이전트 현황 한눈에."

- 에이전트 목록, 상태, 활동 로그
- 행동 지침 편집
- 비용(토큰 사용량) 모니터링

## 6. User Stories

### 시나리오 A: 개발팀 — 코드 리뷰어

**Setup (팀 리드)**
1. Web UI → "새 에이전트" → 이름: "리뷰어"
2. 행동 지침: "PR이 올라오면 팀 코딩 컨벤션에 맞춰 리뷰한다. 심각한 이슈는 Request Changes, 사소한 건 코멘트만."
3. Slack #code-review 채널 연결
4. → 이후 PR 알림이 오면 에이전트가 자동 리뷰

**Daily Use (개발자)**
1. DM: "이 이슈 좀 봐줘 #1234"
2. 에이전트: 이슈 읽고, 관련 코드 파악 → "이 부분 수정하면 될 것 같은데, PR 만들까?"
3. "응" → PR 생성

### 시나리오 B: 마케팅팀 — 콘텐츠 어시스턴트

**Setup**
1. "마케팅 봇" 에이전트 생성
2. 행동 지침: "블로그 초안 작성, SNS 카피 생성, 경쟁사 콘텐츠 모니터링"
3. Slack #marketing 연결

**Daily Use**
1. 멘션: "@마케팅봇 다음주 제품 업데이트 블로그 초안 써줘"
2. 에이전트: 제품 릴리즈 노트 읽고 → 블로그 초안 작성 → 스레드에 공유
3. 팀원이 피드백 → 에이전트가 수정

### 시나리오 C: CS팀 — 고객 응대 보조

**Setup**
1. "CS 도우미" 에이전트 생성
2. 행동 지침: "고객 문의 패턴 분석, FAQ 초안 작성, 에스컬레이션 기준 판단"
3. Slack #cs-team 연결

**Daily Use**
1. CS 담당자: "@CS도우미 이번 주 문의 중 반복되는 패턴 있어?"
2. 에이전트: 티켓 데이터 분석 → "결제 관련 문의가 40% 증가, 주로 환불 프로세스에 대한 질문"
3. "FAQ에 환불 프로세스 항목 추가할까?" → 초안 작성

### 시나리오 D: 관리자 (Monitoring)
1. Web UI 대시보드에서 에이전트 활동 확인
2. 에이전트별 처리 건수, 토큰 사용량
3. 행동 지침 수정 → 즉시 반영

> **Note:** 시나리오 A(개발팀)는 PoC에서 직접 검증. B, C는 가설 — 제품 비전으로 포함하되, 검증은 이후 단계.

## 7. What This Is NOT

- ❌ **Copilot 대체** — IDE 내 코드 자동완성이 아님
- ❌ **CI/CD 파이프라인** — 빌드/배포 자동화가 아님
- ❌ **챗봇** — Q&A에 답하는 봇이 아님 (능동적으로 행동하는 에이전트)
- ❌ **노코드 자동화** — Zapier/n8n처럼 워크플로우를 그리는 게 아님
- ❌ **특정 직군 전용** — 개발자만을 위한 제품이 아님

## 8. Success Metrics

### North Star Metric
**에이전트가 팀원으로서 기여한 작업 수** (완료된 요청, 능동적 행동 / 주)

### Leading Indicators
| Metric | Target (3개월) |
|--------|---------------|
| 일일 활성 사용 (에이전트와 대화) | 팀원 80%+ |
| 에이전트 능동적 행동 / 주 | 10+ |
| 평균 응답 시간 | < 30초 |
| 에이전트 추가 소요 시간 | < 5분 |

### Guardrails
| Metric | Threshold |
|--------|-----------|
| 에이전트 실수율 (사용자가 되돌린 작업) | < 10% |
| 월 비용 / 에이전트 | < $200 |
| 다운타임 | < 1% |

## 9. Technical Approach (Summary)

자세한 아키텍처는 `architecture.md` 참고.

| Component | Choice | Rationale |
|-----------|--------|-----------|
| LLM Runtime | Pi SDK (@mariozechner/pi-*) | OpenClaw 코어, 실전 검증, 도구 호출 내장 |
| Slack | Bolt (TypeScript) | 공식 프레임워크 |
| Auth | setup-token (Claude 구독) | 월정액, API 과금 불필요 |
| Storage | 파일시스템 + SQLite | 심플, 초기 단계에 적합 |
| 기본 도구 | 파일 읽기/쓰기, 셸, 웹 검색/페치 | 최소한의 도구로 시작, 셸로 확장 가능 |
| Web UI | TBD | P1에서 결정 |

### 도구 철학
- 기본 도구는 최소한으로: 파일, 셸, 웹
- 셸이 있으면 git, gh, curl 등 뭐든 실행 가능
- 역할별 전용 도구가 필요하면 그때 만들어 쓴다
- 처음부터 많이 넣지 않는다

## 10. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| setup-token 정책 변경 | 인증 불가 | API 키 방식 병행 지원 |
| LLM 환각으로 잘못된 작업 | 신뢰도 하락 | 위험한 행동은 사람 확인 단계 추가 |
| 토큰 비용 예측 어려움 | 예산 초과 | 에이전트별 일일 한도 설정 |
| Slack rate limit | 응답 지연 | 메시지 큐, 배치 전송 |
| 비개발 시나리오 검증 부족 | 범용성 약속 못 지킴 | PoC는 개발팀, 이후 단계에서 다른 팀 검증 |
| 팀 저항감 | 도입 실패 | "대체"가 아닌 "보조" 포지셔닝, 점진적 도입 |
