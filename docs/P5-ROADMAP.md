# P5 Roadmap — 채널 라우팅 & 슬래시 커맨드

> 목표: 봇 초대 → 즉시 사용 가능한 UX. 설정 파일 수동 편집 제거.

---

## 배경 & 결정사항

- 슬랙봇 1개 + 에이전트 N개 구조 (P3에서 확정)
- 에이전트-채널 매핑을 `agent.json`에서 동적으로 관리
- 스레드 기반 라우팅: 멘션 → 스레드 생성 → 스레드 내 대화 지속
- 컨시어지는 별도 에이전트가 아닌 라우터 fallback + 슬래시 커맨드 핸들러

## Phase 1: 채널 설정 구조 변경

**목표:** 채널별 모드(mention/proactive) 지원

```jsonc
// 기존
"slackChannels": ["C0AHPDVG7EF"]

// 신규
"channels": {
  "C0AHPDVG7EF": { "mode": "mention" },
  "C0BXYZ12345": { "mode": "proactive" }
}
```

- [x] `ChannelConfig` 타입 정의: `{ mode: "mention" | "proactive" }`
- [x] `slackChannels: string[]` → `channels: Record<string, ChannelConfig>`로 스키마 변경
- [x] `findBySlackChannel()` 등 내부 API 업데이트
- [x] Web UI 채널 설정 UI 업데이트

## Phase 2: 컨시어지 (라우터 fallback)

**목표:** 매핑 안 된 채널에서 봇 멘션 시 안내 응답

- [x] 라우터에 fallback 로직: 매핑 없는 채널 → 고정 안내 메시지
- [x] 안내 내용: 사용 가능한 에이전트 목록 + `/assign` 사용법
- [x] LLM 호출 없이 룰 베이스 응답 (토큰 절약)

## Phase 3: 슬래시 커맨드

**목표:** 슬랙 안에서 에이전트-채널 매핑 관리

- [x] `/assign {agent}` — 현재 채널에 에이전트 배정 (agent.json 업데이트)
- [x] `/unassign` — 현재 채널 매핑 해제
- [x] `/agents` — 사용 가능한 에이전트 목록 조회
- [x] `/switch {agent}` — 채널 기본 에이전트 교체
- [ ] 권한 제어: 채널 관리자만 `/assign` 가능 (옵션, 추후)

## Phase 4: 스레드 라우팅 고도화

**목표:** 한 채널에서 여러 에이전트 자연스럽게 사용

- [x] 기본 멘션 → 채널 기본 에이전트로 스레드 생성
- [x] `@봇 /as {agent}` → 특정 에이전트로 스레드 생성
- [x] 스레드 내 대화는 멘션 없이 해당 에이전트가 응답
- [x] 스레드 시작 시 최근 채널 메시지 5~10개를 컨텍스트로 주입

## Phase 5: 응답 적극성 제어 (P4에서 이관)

**목표:** 기본값은 적극적. 유저가 AGENTS.md에서 직접 조절. 스레드/채널 구조 확정 후 구체화.

- [x] 기본 AGENTS.md 템플릿에 DM vs 그룹(채널) 정책 섹션 추가
- [x] 그룹 응답 판단 체크리스트 내장 ("나한테 물어본 건가? 새 정보 있나? 누가 이미 답했나?")
- [x] 과잉 응답 방지 규칙 강화 (중복 응답/잡담 과개입 방지)
- [x] 스레드 내 vs 채널 직접 응답 가이드라인

## Phase 6: Proactive 모드 (미래)

**목표:** 에이전트가 채널을 감시하고 판단해서 끼어드는 모드

- [x] `channelMode: "proactive"` 설정 시 채널 메시지 리스닝
- [x] 끼어들기 판단 로직 (AGENTS.md 가이드라인 기반)
- [x] 끼어들 때 새 스레드로 제안 (채널 오염 방지)
- [x] 토큰 예산 관리 (모든 메시지를 LLM에 보내지 않음)

---

## 실행 흐름 요약

```
[봇 초대] → [매핑 없음] → 컨시어지 안내
                ↓
        /assign engineer
                ↓
[유저 멘션] → [라우터: 채널→에이전트] → [스레드 생성] → [대화]
                ↓
        @봇 /as analyst (다른 에이전트)
                ↓
        [별도 스레드 생성] → [analyst 세션]
```

## 운영 원칙

- 각 Phase는: 구현 → Codex 리뷰 → 빌드/테스트 → 커밋
- 위험 변경(마이그레이션/스키마 변경)은 사전 보고
- 운영 에이전트 없으므로 마이그레이션 불필요 — 스키마 직접 변경
