# P1 Roadmap — Should Have

> P0 (PoC) 완료 기준: 2026-03-02
> P1 목표: MVP로 가기 위한 핵심 기능 구현

---

## 현재 상태 (P0 완료)

### 작동하는 것
- Slack DM → 에이전트 응답
- 채널 @멘션 → 스레드 응답
- 채널 메시지 → 라우팅 + 응답 (message.channels 구독 시)
- 스레드 후속 메시지 → 같은 세션 유지
- Pi SDK 기반 에이전트 세션 (파일, 셸, 웹 도구)
- 에이전트별 워크스페이스 (AGENTS.md, agent.json)
- 세션 idle timeout + 자동 정리
- REST API (에이전트 목록, 상태, 메시지 전송)

### 빠져 있는 것
- Web UI 없음 (API만 있음)
- 에이전트 생성 UI 없음 (수동으로 폴더 + agent.json 생성)
- 스케줄러/하트비트 없음 (능동적 행동 불가)
- 비용 트래킹 없음
- 파일/이미지 첨부 처리 안 됨
- 이모지 리액션 미구현
- 에이전트 로그/활동 기록 없음
- 복수 에이전트 동시 운영 미검증

## P1 구현 목록

### 1. Web UI (기본)

**목표:** 에이전트를 코드 수정 없이 관리할 수 있는 최소 인터페이스

| 항목 | 설명 | 우선순위 |
|------|------|----------|
| 에이전트 목록 | 이름, 상태(활성/비활성), 마지막 활동 시간 | 필수 |
| 에이전트 상세 | agent.json + AGENTS.md 내용 표시 | 필수 |
| 에이전트 생성 | 이름 → 역할 지침 → Slack 채널/사용자 매핑 → 생성 | 필수 |
| 에이전트 수정 | agent.json 필드 편집, AGENTS.md 텍스트 에디터 | 필수 |
| 에이전트 활성/비활성 | enabled 토글 | 필수 |
| 활동 로그 | 최근 메시지/응답 타임라인 | 선택 |

**기술 결정 필요:**
- 프레임워크: React? Next.js? HTMX? SvelteKit?
- 기존 API 서버(port 3001)에 정적 파일 서빙 vs 별도 프론트

---

### 2. 에이전트 생성 플로우

**목표:** "5분 안에 새 에이전트 추가"

- `POST /api/agents` — 워크스페이스 폴더 + agent.json + AGENTS.md 템플릿 생성
- agent.json 필수 필드 검증
- Slack 채널/사용자 ID 자동 조회 (채널 이름 → ID 변환)
- `DELETE /api/agents/:id` — 에이전트 삭제
- **핫 리로드**: 에이전트 추가/수정 시 서버 재시작 없이 레지스트리 갱신

---

### 3. 스케줄러 / 하트비트

**목표:** 에이전트가 주기적으로 할 일을 확인하고 행동

| 항목 | 설명 |
|------|------|
| 하트비트 루프 | 설정된 주기마다 에이전트 세션에 heartbeat prompt 전달 |
| HEARTBEAT.md | 에이전트별 체크리스트 — 하트비트 시 읽고 실행 |
| 크론 잡 | 특정 시간에 특정 작업 실행 |
| 조용한 하트비트 | 할 일 없으면 Slack 메시지 안 보냄 |

agent.json 확장:
```json
{
  "heartbeat": {
    "enabled": true,
    "intervalMin": 30,
    "quietHours": [23, 8]
  },
  "cron": [
    {
      "name": "daily-report",
      "schedule": "0 9 * * *",
      "prompt": "일일 리포트를 #general에 공유해줘"
    }
  ]
}
```

### 4. 비용 트래킹

**목표:** 에이전트별 토큰 사용량 + 비용 추정

| 항목 | 설명 |
|------|------|
| 토큰 카운트 | Pi SDK 응답에서 usage 추출 → SQLite 저장 |
| 에이전트별 집계 | 일별/주별/월별 토큰 사용량 |
| 비용 추정 | 모델별 가격표 기반 |
| 일일 한도 | 에이전트별 토큰 상한 → 초과 시 경고 or 중지 |
| API | `GET /api/agents/:id/usage` |

---

### 5. Slack 인터랙션 강화

| 항목 | 설명 | 난이도 |
|------|------|--------|
| 파일/이미지 첨부 수신 | files 필드 파싱 → 다운로드 → 에이전트에 전달 | 중 |
| 파일 업로드 (봇→Slack) | 에이전트가 생성한 파일을 Slack에 공유 | 중 |
| 이모지 리액션 수신 | reaction_added 이벤트 구독 + 처리 | 하 |
| 봇 리액션 달기 | 처리 시작 시 👀, 완료 시 ✅ 등 상태 피드백 | 하 |
| 타이핑 인디케이터 | 응답 생성 중 typing 표시 | 하 |
| 마크다운 → Block Kit | 응답을 Slack Block Kit으로 포맷팅 | 중 |
| 스레드 자동 생성 | 채널 메시지에 항상 스레드로 응답 | 하 |
| 에러 핸들링 | LLM 실패 시 사용자에게 친절한 메시지 | 하 |

**Event Subscriptions 추가 필요:**
- `reaction_added`
- `file_shared` (또는 message event의 files 필드 활용)

| 항목 | 설명 |
|------|------|
| 라우팅 개선 | 채널에 여러 에이전트 매핑 시 멘션 기반 disambiguation |

| 동시 세션 | 여러 에이전트 × 여러 사용자 동시 대화 |
| 리소스 관리 | 에이전트별 동시 세션 수 제한 |

## 구현 순서 (제안)

```
Phase 1 (1주): 기반
├── 핫 리로드 (레지스트리 파일 감시)
├── 에이전트 CRUD API
├── Slack 리액션 (👀 처리중, ✅ 완료)
├── 에러 핸들링 + 사용자 메시지
└── 디버그 로깅 → INFO

Phase 2 (1주): Slack 강화
├── 파일/이미지 첨부 수신
├── 파일 업로드 (봇→Slack)
├── 마크다운 → Block Kit
├── 타이핑 인디케이터
└── 스레드 자동 생성

Phase 3 (1주): 능동성
├── 하트비트 루프
├── HEARTBEAT.md 처리
├── 크론 잡 스케줄러
└── 조용한 하트비트

Phase 4 (1주): 관리
├── Web UI (목록 + 생성 + 편집)
├── 비용 트래킹 (토큰 → SQLite)
├── 헬스체크 API
└── PM2 설정
```

## 기술 결정 보류 항목

| 항목 | 선택지 | 결정 시점 |
|------|--------|----------|
| Web UI 프레임워크 | React / Next.js / HTMX / SvelteKit | Phase 4 전 |
| DB | SQLite vs PostgreSQL | 부하 테스트 후 |
| 인증 | 없음 / Basic Auth / OAuth | 외부 배포 시 |
| 파일 저장 | 로컬 FS vs S3 | 스케일 필요 시 |
